Real-World Implementation Guide

A step-by-step technical breakdown of how to build the "Infectious Disease Spread Simulation Based on Large Language Model Decision Making" paper at scale.

0. The Architecture at Scale

While browser-based simulations are great for visualization, reproducing the paper's methodology for millions of agents requires a robust, distributed backend architecture. Here is the high-level system flow:

1. Synthetic Population Generation

The first step is to create a realistic population that mirrors the demographics of a specific city (like Atlanta or San Francisco) using real-world census data.

  • Data Source: US Census Bureau API (TIGER/Line shapefiles for geographical boundaries and demographic tables).
  • Methodology: Use conditional probability matching. Instead of random sampling, you generate agents whose attributes (Age, Race, Gender, Education, Income) match the exact distribution of their specific census tract.

2. Pre-computing the LLM Decision Bank

Querying an LLM for every single agent's decision during a real-time simulation is computationally impossible and cost-prohibitive. The paper's critical innovation is the pre-computed decision bank.

You generate all possible demographic combinations (e.g., 2 Age groups × 4 Races × 2 Genders × 3 Education levels × 2 Income levels = 96 base profiles). You then query the LLM multiple times for each profile to get a probability distribution of their willingness to self-report.

# Generating the Decision Bank via LLM API (e.g., vLLM or OpenAI)
async def generate_decision_bank(profiles, prompts, llm_client):
    decision_bank = {}
    
    for profile in profiles:
        for prompt_scenario in prompts:
            responses = await llm_client.generate(
                prompt=build_prompt(profile, prompt_scenario),
                n=5, # Sample 5 times to get a probability
                temperature=0.6
            )
            
            # Count 'Yes' responses to calculate probability
            yes_count = sum(1 for r in responses if 'Yes' in r.text)
            key = f"{profile.hash}_{prompt_scenario.id}"
            
            decision_bank[key] = yes_count / 5.0
            
    return decision_bank

3. The SEIR Disease & Simulation Engine

With the population and decision bank ready, the simulation engine runs in discrete time steps (ticks). The paper uses an extended SEIR (Susceptible, Exposed, Infectious, Recovered) model.

stateDiagram-v2 Susceptible --> Exposed : Physical Contact Exposed --> Infectious : Incubation Period Infectious --> Symptomatic : Probability P(sym) Infectious --> Asymptomatic : Probability 1-P(sym) Symptomatic --> LLM_Decision : Query Decision Bank LLM_Decision --> Isolated : Yes (Reported) LLM_Decision --> Infectious : No (Unreported) Isolated --> Recovered : Recovery Period Asymptomatic --> Recovered : Recovery Period

In a distributed backend (like Java or Python with multi-processing), spatial hashing (like QuadTrees) is used to efficiently calculate which agents come into contact with each other in the simulated "Patterns of Life" (moving between home, work, and shops).

4. Execution and Output Tracking

As the simulation runs, you track the state transitions of every agent. The critical output is comparing the true infection curve against the reported infection curve.

  • Because the LLM decision bank applies demographic biases (e.g., lower income agents report less due to economic pressure), the reported data will visually diverge from the true data in specific census tracts.
  • Save the tick-by-tick results to a scalable format like Parquet for downstream data science analysis in Jupyter Notebooks.
